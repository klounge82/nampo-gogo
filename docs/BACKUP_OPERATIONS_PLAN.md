# Nampo GoGo 데이터 백업 및 복구 운영 계획서 (BACKUP_OPERATIONS_PLAN)

본 문서는 Nampo GoGo 서비스의 데이터 무손실 보장(RPO 24시간) 및 장애 복구 시간(RTO 4시간) 목표 달성을 위한 백업 정책 및 운영 계획입니다.

---

## 1. 백업 보관 정책 (Retention Policy)
- **일일 백업 (Daily Backup)**: 매일 새벽 3시에 수행하며, **최근 7일 분량** 보관.
- **주간 백업 (Weekly Backup)**: 매주 일요일 새벽 3시에 수행하며, **최근 4주 분량** 보관.
- **데이터 백업 대상**: PostgreSQL 데이터베이스 전체 스키마 및 레코드 데이터 덤프(`.dump` 파일).
- **RPO 목표 (Recovery Point Objective)**: **최대 24시간** 이내의 데이터 유실 한도 설정.
- **RTO 목표 (Recovery Time Objective)**: 장애 발생 시 **최대 4시간** 이내에 서비스를 복구 완료함.

---

## 2. 백업 파일 저장 위치 및 상태

| 위치 및 요소 | 상세 요건 | 현재 상태 | 개선 요구사항 |
| :--- | :--- | :--- | :--- |
| **백업 구동 서버** | 백엔드 WAS 가 돌아가는 기동 머신 | **AVAILABLE** | `db_backup.sh` 쉘 구현 완료 |
| **로컬 보관 디렉토리**| `/var/backups/nampogogo` 내부 | **AVAILABLE** | 스크립트 기본 경로 매핑 완료 |
| **격리 저장소 (Off-site)** | AWS S3, Google Cloud Storage 등 외부 스토리지 | **MISSING** | 서버 폭발 시 대비한 클라우드 버킷 백업 이관 필요 |
| **암호화 여부** | 백업 덤프 파일 내 민감정보 암호화 보관 여부 | **MISSING** | 대칭키 암호화(gpg) 도입 권장 |
| **자동 크론 설정** | `0 3 * * *` 스케줄러 매핑 | **MISSING** | 실제 서버 crontab 에 스크립트 매핑 필요 |

---

## 3. 백업 실패 알림 및 모니터링
- **스크립트 수준 감지**: `db_backup.sh` 구동 시 에러 코드가 리턴되거나, 생성된 백업 파일의 크기가 0바이트인 경우 실패 로그 적재 및 즉시 `exit 1` 처리.
- **모니터링 알림**: 크론 구동 실패 시 Discord webhook, Slack, 또는 이메일을 통해 시스템 관리자에게 실시간 경보 메시지가 발송되도록 셋업 권장.

---

## 4. 주기적 복구 테스트 (Restore Test Plan)
- **테스트 주기**: **매 분기 1회** (3개월 주기)
- **테스트 방법**: 개발 스태이징 서버 또는 임시 샌드박스 DB에 최근 백업 파일을 `db_restore.sh` 로 밀어 넣은 뒤, 메인 로그인 및 매장 조회 API 가 에러 없이 작동하는지 교차 검사 실행.

---

## 5. 비상 대응 및 담당 역할
- **비상 대응 담당자**: 시스템 운영 총괄 (Operations Owner)
- **책임 범위**: 백업 스크립트 주기 점검, 격리 저장소 보안 관리, 장애 복구 상황 발생 시 `db_restore.sh` 원격 수행 및 롤백 절차 통제.
- **주의**: 로컬 개발 머신의 격리되지 않은 소스 폴더를 운영 백업 위치로 오용하는 행위를 일체 엄금합니다.
