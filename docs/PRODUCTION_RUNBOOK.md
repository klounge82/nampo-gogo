# 남포 GoGo Production 장애 대응 Runbook

본 문서는 남포 GoGo 서비스 운영 중 장애(Liveness/Readiness 실패, 5xx 급증, 결제 실패 등) 발생 시 신속하고 정합하게 복구하기 위한 운영 지침서입니다.

---

## 1. 장애 등급 정의

| 장애 등급 | 영향 범위 | 대응 목표 시간 | 예시 상황 |
| :--- | :--- | :--- | :--- |
| **SEV-1** | 전체 서비스 중단, DB 손상, 개인정보/Secret 누출, 중복 결제 발생 | 즉시 확인, 30분 이내 조치 시작 | FastAPI 서버 크래시, DB 연결 영구 실패, JWT_SECRET 평문 노출 |
| **SEV-2** | 로그인 불가, 지도/QR/예약 등 핵심 기능 마비, 대규모 5xx 에러, 백업 연속 실패 | 1시간 이내 대응 시작 | FCM 토큰 전송 전체 차단, 일일 백업(db_backup.sh) 2회 이상 연속 실패 |
| **SEV-3** | 일부 기능 오류, 특정 사용자군/매장 한정 장애, 이미지 업로드 실패 | 업무시간 내 확인 및 대응 | 프로필 이미지 업로드 실패, 특정 매장 리뷰 평점 집계 지연 |
| **SEV-4** | 경미한 UI 오작동, 비핵심 기능 마이너 오류 (우회 경로 존재) | 정기 배포 일정에 반영 | 오프라인 배너 문구 오타, 일부 통계 대시보드 렌더링 딜레이 |

---

## 2. 장애 전파 및 비상 연락 체계

장애 발생 시 하드코딩된 연락처를 사용하지 않고, 다음 역할 분담을 준수하여 보고를 전파합니다:
- **운영 책임자 (Service Owner)**: 외부 공지 여부 결정, 유관 부서 전파 총괄.
- **Backend 담당자 (Tech Lead)**: Request ID 로그 분석, 복원/롤백 스크립트 실행, DB 상태 점검.
- **고객지원 담당자 (CS Lead)**: 사용자 민원 인입 모니터링, 장애 안내 팝업 메시지 송출 조율.
- **최고 관리자 (Super Admin)**: 시스템 긴급 계정 잠금 및 인프라 강제 복원 승인.

## 3. 장애 감지 및 초기 대응 절차

### 1단계: Health Check 및 인프라 생존 확인
- **Liveness API**: `GET /health/live` 응답이 200 OK 인지 점검. (프로세스 생존 여부)
- **Readiness API**: `GET /health/ready` 응답이 200 OK 인지 점검.
  - 만약 503 Service Unavailable 및 `database: unavailable` 상태가 보인다면, 백엔드가 DB 와 통신하지 못하고 있는 상태입니다.

### 2단계: Request ID 연계 로그 추적
- 사용자 응답이나 Nginx 액세스 로그에 찍힌 `X-Request-ID` 헤더 값을 추출합니다.
- 서버 터미널에서 해당 ID를 검색하여 Exception Stack Trace와 Context를 조회합니다:
  ```bash
  grep "ID:<request_id>" /var/log/nampogogo_access.log
  ```
- **개인정보보호 경고**: 마스킹된 로그(`****REDACTED****`) 외에 날것의 비밀번호나 토큰을 로그 파일에 직접 기록하지 마십시오.

### 3단계: 임시 우회 및 복구 조치
- 데이터베이스 락업이나 슬로우 쿼리 폭주 시, DB 세션을 정리하거나 Gunicorn 프로세스를 리스타트합니다.
- 복구가 지연될 시, 프록시(Nginx) 단에서 임시 점검 페이지 화면을 유저에게 노출합니다.
